# Лабораторная работа №4: Интерфейсы и абстрактные классы (ABC)

## Цель работы
- Познакомиться с абстрактными базовыми классами (ABC) и модулем `abc`.
- Освоить понятие интерфейса как контракта поведения.
- Научиться задавать обязательные методы для классов с помощью `@abstractmethod`.
- Закрепить полиморфизм через единый интерфейс.
- Научиться проектировать архитектуру, а не просто отдельные классы.

## Постановка задачи
На основе системы, разработанной в ЛР1–ЛР3 (базовый класс `Student`, наследники `BakalavrStudent` и `MagaStudent`, коллекция `StudentCollection`), необходимо ввести контракты поведения, которые обязывают классы реализовать определённые методы. Контракты должны быть оформлены в виде абстрактных базовых классов (интерфейсов). Требуется продемонстрировать полиморфную работу с объектами через эти интерфейсы, а также интегрировать интерфейсы в коллекцию для фильтрации и сортировки.

## Требования и реализация (уровень 5)

### 1. Интерфейсы (ABC)
- Создано два интерфейса: `Printable` и `Comparable`.
- В каждом интерфейсе присутствует минимум один абстрактный метод:
  - `Printable.to_string() -> str` – возвращает строковое представление объекта.
  - `Comparable.compare_to(other) -> int` – сравнивает текущий объект с другим; возвращает отрицательное число, ноль или положительное число.
- Оба интерфейса наследуют от `ABC` и используют декоратор `@abstractmethod`.

### 2. Реализация интерфейсов в классах
- Классы `Student`, `BakalavrStudent` и `MagaStudent` расширены – теперь они реализуют оба интерфейса.
- `to_string()` во всех классах делегирует вызов существующему методу `__str__()`, обеспечивая единый способ получения строкового представления.
- `compare_to()` реализован в каждом классе по-своему:
  - Обычный студент сравнивается по среднему баллу (GPA).
  - Бакалавр – сначала по курсу, затем по GPA.
  - Магистрант – с другими магистрантами сначала по наличию утверждённой темы диссертации, затем по GPA; с остальными – по курсу и GPA.
- Дополнительно продемонстрировано поведение объекта, не реализующего интерфейсы (оригинальный `Student` из ЛР3).

### 3. Использование интерфейса как типа
- Универсальная функция `print_all(items: list[Printable])` принимает список любых объектов, реализующих `Printable`, и выводит их строковое представление, не зная конкретного типа.
- Аналогичная функция `show_sorted_comparables(items: list[Comparable])` сортирует переданные объекты, используя их собственный метод `compare_to()`.

### 4. Множественная реализация интерфейсов
- Все три класса (`Student`, `BakalavrStudent`, `MagaStudent`) одновременно являются и `Printable`, и `Comparable`. Это подтверждается проверками `isinstance()` во время демонстрации.

### 5. Интеграция с коллекцией (ЛР-2)
- Класс `ExtendedStudentCollection` унаследован от существующей `StudentCollection` и дополнен методами:
  - `get_printable() -> list[Printable]` – возвращает все элементы коллекции, реализующие `Printable`.
  - `get_comparable() -> list[Comparable]` – возвращает все элементы, реализующие `Comparable`.
  - `get_sorted_comparables(reverse=False) -> list[Comparable]` – возвращает список `Comparable`-объектов, отсортированных исключительно через вызов их метода `compare_to()` (с использованием `functools.cmp_to_key`).

### 6. Полиморфизм без явных проверок типа
Вся бизнес-логика (вывод, сортировка) построена на вызове интерфейсных методов. Проверки `isinstance` используются только в методах коллекции для фильтрации, но не в алгоритмах обработки данных 

## Сценарии 
### Сценарий 1: Полиморфный вывод через Printable
Создаётся коллекция, в которую добавляются пять студентов разных типов (обычный, два бакалавра, два магистранта).  
У коллекции вызывается метод `get_printable()`, который возвращает список всех объектов, так как каждый из них реализует `Printable`.  
Затем универсальная функция `print_all()` проходит по этому списку и для каждого элемента вызывает `to_string()`.  
Результат: на экран выводятся строковые представления всех студентов, причём для каждого типа используется своя реализация `to_string` (бакалавр выводит специализацию и практику, магистрант – научного руководителя и тему диссертации).  
Этот сценарий доказывает, что можно единообразно работать с разнородными объектами, не привязываясь к их конкретным классам.

### Сценарий 2: Сортировка через Comparable
Используется метод коллекции `get_sorted_comparables()`. Он извлекает все `Comparable`-объекты и сортирует их с помощью компаратора, построенного на основе их собственного метода `compare_to()`.  
Поскольку каждый тип задаёт свои правила сравнения, порядок сортировки получается осмысленным именно для данной предметной области: бакалавры упорядочиваются сначала по курсу, затем по успеваемости; магистранты – по наличию темы и баллам.  
На экран выводятся отсортированные студенты. Сценарий демонстрирует архитектурное поведение: алгоритм сортировки не знает о полях `gpa` или `thesis_topic`, он лишь вызывает контрактный метод `compare_to`.

### Сценарий 3: Работа с объектом без интерфейсов
В коллекцию добавляется «старый» объект `Student`, импортированный напрямую из `lab03.base`, который не наследует ни `Printable`, ни `Comparable`.  
После этого:
- `get_printable()` возвращает только тех студентов, которые реализуют интерфейс; старый объект игнорируется.
- При ручном обходе всей коллекции с проверкой `isinstance(obj, Printable)` мы видим, что старый студент не поддерживает `Printable`, и для него выводится замещающее сообщение.
- Сортировка через `get_sorted_comparables()` также не затрагивает этот объект, поскольку он не попадает в выборку `Comparable`.
Таким образом, интерфейсы служат не только контрактом, но и маркером возможностей объекта; коллекция корректно разделяет объекты по их способностям.

### Сценарий 4: Универсальная функция и полиморфизм
Функция `show_sorted_comparables()` принимает список `Comparable` (любых объектов, поддерживающих сравнение) и выполняет сортировку, ничего не зная о внутреннем устройстве элементов.  
Она демонстрирует, что новый тип, реализующий `Comparable`, будет автоматически поддерживаться без изменения кода функции.  
В рамках демонстрации функция вызывается для списка всех `Comparable`-студентов, и на экран выводятся отсортированные данные.

## Польза введённых интерфейсов
- **Разделение ответственности.** Классы отвечают только за свою реализацию контракта; код, использующий их, зависит не от конкретных полей, а от абстрактных методов.
- **Гарантия поведения.** Благодаря `@abstractmethod` невозможно создать экземпляр класса, забывшего реализовать `to_string` или `compare_to`.
- **Расширяемость.** При добавлении нового типа студента (например, `Aspirant`) коллекция и универсальные функции продолжат работать без модификации.
- **Чистый полиморфизм.** Условные операторы `if isinstance(...)` ушли из бизнес-логики; решения принимаются самими объектами через переопределённые методы.
