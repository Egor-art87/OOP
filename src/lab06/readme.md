# Лабораторная работа №6 — Generics и typing

## Цель работы

Целью данной лабораторной работы является освоение системы аннотаций типов в языке Python, в том числе создание обобщённых (generic) классов с использованием TypeVar и Generic, а также понимание концепции структурной типизации через typing.Protocol. В процессе выполнения необходимо было расширить существующую коллекцию объектов, созданную в предыдущих работах, сделав её типизированной и гибкой для работы с различными типами данных без нарушения уже написанного кода.

## Ход мыслей при написании работы

Приступая к выполнению задания, я руководствовался необходимостью постепенно наращивать возможности ранее созданного кода, не ломая его. В лабораторных работах №1–3 были разработаны базовый класс Student, его наследники BakalavrStudent и MagaStudent, а также контейнер StudentCollection, хранящий объекты этих классов. Контейнер был удобен, но не обладал информацией о типе хранимых элементов на уровне аннотаций, что затрудняло статический анализ и могло приводить к ошибкам при расширении.

Первым шагом стало создание отдельного модуля container.py, в котором я описал обобщённый класс TypedCollection[T]. Для этого был использован TypeVar без ограничений, что позволило параметризовать коллекцию любым типом. Все методы оригинальной StudentCollection были перенесены с добавлением корректных аннотаций типов: параметры методов, возвращаемые значения и атрибуты получили соответствующие типы, связанные с T.

Далее для обеспечения функциональности высшего порядка (передача функций-аргументов) были добавлены методы find, filter и map. Их сигнатуры потребовали использования Callable из модуля typing. Поскольку метод map может преобразовывать элементы в значения другого типа, был введён второй TypeVar с именем R, что позволило правильно аннотировать возвращаемый список list[R]. Таким образом, статический анализатор теперь понимает, что тип результата map может отличаться от типа элементов коллекции.

Ключевой идеей работы на высокую оценку стала демонстрация структурной типизации через протоколы. Я определил два протокола: Displayable, требующий наличия метода display() -> str, и Scorable, требующий метода score() -> float. Затем были созданы ограниченные TypeVar: D с bound=Displayable и S с bound=Scorable. Это позволило объявлять коллекции, которые принимают только объекты, удовлетворяющие соответствующему протоколу, и внутри такой коллекции можно безопасно вызывать методы протокола. При этом классы предметной области не были изменены — я лишь создал небольшие наследники в demo.py, которые добавляют нужные методы, и эти наследники автоматически считаются совместимыми с протоколами без явного наследования от них. Такой подход продемонстрировал гибкость утиной типизации на уровне статического анализа.

В итоге все компоненты были собраны в demo.py, где реализованы два сценария, показывающих работу коллекции с разными ограничениями типов. Это позволило наглядно убедиться, что аннотации не влияют на выполнение программы, но дают возможность IDE и статическим анализаторам выявлять потенциальные ошибки.

## Описание сценариев

### Сценарий 1 — Коллекция с ограничением Displayable

В первом сценарии создаётся экземпляр TypedCollection, параметризованный типом, удовлетворяющим протоколу Displayable. Это означает, что компилятор (mypy/Pyright) ожидает от элементов наличия метода display(), возвращающего строку. В коллекцию добавляются три объекта: два бакалавра и один магистр, созданные на основе классов DisplayableBachelor и DisplayableMaster, которые наследуют оригинальных студентов и реализуют необходимый метод. После добавления происходит итерация по коллекции с вызовом display() у каждого элемента — выводятся строки с информацией об имени, курсе и GPA. Далее демонстрируются новые методы коллекции. Метод find с предикатом, проверяющим GPA больше 4.6, находит первого подходящего студента (это оказывается Diana) и выводит её представление через display(); если бы такого студента не было, метод вернул бы None. Метод filter отбирает всех студентов, курс которых не меньше второго, и для каждого выводится результат display(). Наконец, метод map применяет лямбда-функцию, формирующую строку с именем и курсом, и возвращает список строк, который затем печатается. Таким образом, показано, что map меняет тип результата с Displayable на str, и коллекция корректно работает с предикатами и функциями преобразования.

### Сценарий 2 — Коллекция с ограничением Scorable

Во втором сценарии используется тот же набор объектов, но коллекция объявляется с ограничением Scorable. Протокол Scorable требует наличия метода score(), возвращающего вещественное число. Объекты, созданные ранее, содержат такой метод (для бакалавров он возвращает GPA, для магистров — GPA с бонусом), поэтому они успешно добавляются в новую коллекцию. После добавления выполняется обход элементов с выводом результата display() (для наглядности) и значения score(). Затем метод filter отбирает элементы с оценкой больше 4.0, и для каждого выводится его строковое представление и значение score(). В завершение метод map с функцией, извлекающей score, возвращает список значений типа float, который выводится на экран. Здесь особенно хорошо видно, что map действительно меняет тип результата: на входе были объекты, соответствующие Scorable, а на выходе — список чисел с плавающей точкой. Этот сценарий подчёркивает, что один и тот же класс TypedCollection может быть использован с разными протоколами, и ограничения bound обеспечивают безопасный вызов методов протокола.

## Общий вывод по работе

Выполнив лабораторную работу, я получил практические навыки использования современной системы типов Python. Аннотации типов, обобщённые классы и протоколы открывают путь к написанию более надёжного, самодокументированного и легко поддерживаемого кода. Статический анализ типов помогает обнаруживать ошибки на ранних этапах разработки, а структурная типизация через Protocol позволяет описывать интерфейсы, не привязываясь к конкретным классам, что особенно ценно при расширении существующих систем. Методы высшего порядка find, filter и map, аннотированные с помощью Callable и дополнительного TypeVar, демонстрируют гибкость функционального подхода в сочетании с типобезопасностью. В целом работа подтвердила, что даже в динамически типизированном языке можно эффективно применять концепции статической типизации, повышая качество и читаемость программного продукта.
