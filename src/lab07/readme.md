# Лабораторная работа №7: Консольное приложение

## 1. Цель работы
Объединить наработки, полученные в рамках ЛР1–ЛР6, в единое полноценное консольное приложение (CLI). Главная задача — реализовать слоистую архитектуру, при которой ввод/вывод изолирован от бизнес-логики и управления данными, а также добавить механизмы сохранения состояния программы.

## 2. Ход мыслей при разработке (Архитектура)
Разработка строилась по принципу "от данных к интерфейсу" для обеспечения строгой изоляции слоёв (Clean Architecture):
1. **Фундамент (Слой ошибок):** В первую очередь были созданы собственные классы исключений (`exceptions.py`), чтобы программа не падала с системными ошибками, а мягко перехватывала логические нестыковки предметной области.
2. **Слой данных (`storage.py`):** Так как коллекция содержит полиморфные объекты, был разработан механизм сериализации и десериализации. Для корректного восстановления объектов из JSON в словари было введено маркерное поле `__type__` (bachelor/master).
3. **Бизнес-логика (`app.py`):** Создан класс `StudentApp`, который инкапсулирует коллекцию. Он стал единственной точкой входа для любых манипуляций с данными, полностью скрыв прямую работу со списками от интерфейса.
4. **Интерфейс (`cli.py`):** Реализован бесконечный цикл `while True`, табличный форматированный вывод и защита "от дурака" с помощью конструкций `try-except`.
5. **Сборка (`main.py`):** Написана короткая точка входа, связывающая бизнес-логику и CLI, реализующая автоматическую загрузку данных при запуске.

## 3. Что использовано из предыдущих лабораторных (ЛР1–ЛР6)
Проект строго базируется на ранее созданной предметной области ("Студенты"):
* **ЛР1–ЛР4 (ООП):** Базовый класс `Student` и его наследники `BakalavrStudent`, `MagaStudent`. Активно используются механизмы инкапсуляции (@property) и полиморфизма (переопределение `calculate_scholarship`, `display_info`).
* **ЛР2, ЛР6 (Коллекции и типизация):** Класс `StudentCollection` (`TypedCollection`) применяется для безопасного хранения и управления списком объектов. Сохранена строгая аннотация типов.
* **ЛР5 (Стратегии):** Функции-стратегии из модуля `strategies.py` (например, `by_name`, `by_gpa_desc`) подключены для сортировки коллекции, также используются методы фильтрации.
* **Пакет lib (Валидация):** Функции `validate_name`, `validate_age`, `validate_course` и др. используются для жесткой проверки данных ещё на этапе инстанцирования объектов.

## 4. Что появилось нового в ЛР7
* **Интерактивный CLI:** Полноценное консольное меню с поддержкой ввода пользователя и циклическим выполнением команд.
* **Безопасный ввод:** Полный перехват ошибок ввода (например, перехват `ValueError` при вводе букв вместо цифр для ID или возраста).
* **Пользовательские исключения (`exceptions.py`):** Реализованы специфичные для проекта ошибки: `StudentNotFoundError`, `DuplicateStudentError`, `InvalidInputError`.
* **Автосохранение и загрузка (`storage.py`):** Реализована персистентность. Данные автоматически загружаются при старте и сохраняются в `students_data.json` при корректном выходе (пункт меню 0).
* **Защита деструктивных действий:** Введен запрос на подтверждение `(y/n)` при попытке удалить студента из базы.
* **Табличный вывод:** Данные коллекции теперь форматируются в аккуратную текстовую таблицу (с фиксированной шириной столбцов) для удобного визуального восприятия.

## 5. Демонстрация работы
В прикрепленной ниже записи терминала продемонстрированы 4 ключевых сценария работы приложения:
1. **Сценарий 1:** Запуск программы, успешная автозагрузка данных из JSON и табличный вывод коллекции.
2. **Сценарий 2:** Попытка фильтрации с некорректным вводом (демонстрация перехвата ошибки) и последующая успешная фильтрация по GPA.
3. **Сценарий 3:** Удаление объекта с подтверждением операции, выход с автосохранением и повторный запуск для проверки сохранения измененного состояния.
4. **Сценарий 4:** Демонстрация сортировки коллекции по выбранной стратегии.
